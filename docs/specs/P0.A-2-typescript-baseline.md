## 1. 背景

P0.A-1 让仓库能 `pnpm install`，但所有子包的 `typecheck` 脚本目前还是 `echo skip`。本 spec 把 TypeScript 真正接入：

- 在 `@aiclassroom/config` 里放**唯一的** `tsconfig.base.json`（项目的 TS 真相源）
- 每个子包和 `apps/web` 有自己的 `tsconfig.json`，`extends` 这份 base
- 子包 `typecheck` 脚本从 `echo skip` 换成 `tsc --noEmit`
- `pnpm -r typecheck` 全绿是本任务的完成标志

**为什么有独立的 base config 包而不是写在根 `tsconfig.json`**：

- 子包未来会有不同的 `target` / `lib`（例如 `apps/web` 需要 DOM，`packages/agents` 只需 Node）。base 提供默认值，子包按需覆盖。
- `apps/web` 里 Next.js 会维护自己的 `tsconfig.json`（P0.C-1 会加），它也 `extends` 同一份 base，保持全仓一致。
- tsconfig 的 `extends` 只能指向**具体文件**，不能指向包名直接写 `"@aiclassroom/config"`；必须写相对路径或 `"@aiclassroom/config/tsconfig.base.json"`（后者要求 `config` 包声明 `exports`）。本 spec 选**相对路径方案**，因为更简单、报错更友好。

**现状**：`packages/config/tsconfig.base.json` 已经存在（P0.A-1 评审时保留未动），字段已相当合理。本 spec 会审计它，补齐缺失字段（如 `moduleDetection`），其余保留。

## 2. 用户故事

- 作为开发者，我在 `packages/types` 写一段有类型错误的代码，执行 `pnpm -r typecheck` **会失败**并指出错误位置 —— 而不是让错误溜进后续任务。
- 作为贡献者，我改 `tsconfig.base.json` 的一个字段（比如加严一条规则），**所有子包**立刻生效，不用到每个包里同步。
- 作为 AI Coding Agent，我要写任何 TS 文件前都能先跑 `pnpm -r typecheck` 得到可预测的基线结果。

## 3. 范围

### 3.1 In Scope

- 审计并完善 `packages/config/tsconfig.base.json`
- 在 7 个子包（`packages/{types,ui,prompts,agents,llm,tts}` + `apps/web`）各自新建 `tsconfig.json`，extends base
- `packages/config` 自身**不需要** `tsconfig.json`（它只是配置容器，没有 TS 源码；由 `.gitattributes` 或 README 备注说明）
- 把 **6 个 `packages/*`**（不含 config）的 `typecheck` 脚本从 `echo skip` 改为 `tsc --noEmit`
- **`apps/web` 的 typecheck 脚本暂时保留 `echo skip`**（见下方说明），`tsconfig.json` 仍要建好供 P0.C-1 接手
- 引入一个"最小测试用例"：在 `packages/types/src/index.ts` 替换掉 `export {};`，改为**一个真实的类型导出**（`export type Placeholder = { readonly _brand: "placeholder" };`），确保 `tsc --noEmit` 真的在跑而不是空过
- `pnpm -r typecheck` 全绿

**为什么 apps/web 暂不切 tsc**：`apps/web/src/` 目前是三个空目录（`app/` / `features/` / `stores/`），没有任何 `.ts` 文件。`tsc` 遇到空 include 会报 `TS18003: No inputs were found`，这是 TypeScript 的设计行为，无法通过配置关闭。解决方案有三：

1. 加一个"placeholder" `.ts` 文件 — 会污染后续 P0.C-1 的 Next.js 目录结构
2. 直接从 workspace 排除 web 的 typecheck — 破坏统一性
3. **apps/web.typecheck 暂时保留 `echo skip`，等 P0.C-1 有真实 TS 源码后切换** — 本 spec 选此方案

这个选择被显式记录在 § 8 AC-6、§ 9 Q4，P0.C-1 的 spec 必须把切换纳入其 In Scope。

### 3.2 Out of Scope

- ESLint 配置 → P0.A-3
- Vitest → P0.A-4
- `packages/types` 里的真实领域类型（Lesson/Scene/Action/...）→ P0.B-1
- TypeScript Project References / incremental build cache — 目前项目规模不需要，等编译慢到烦人再开 spec
- 为 `apps/web` 添加 React/Next.js 类型（`@types/react`, `@types/node`）→ P0.C-1 接入 Next.js 时一起做
- `packages/ui` 的 React 类型同上，推迟到引入 React 代码时再加依赖

## 5. 关键文件内容

### 5.1 `packages/config/tsconfig.base.json`（审计 + 补字段）

目标完整内容：

```jsonc
{
  "$schema": "https://json.schemastore.org/tsconfig",
  "compilerOptions": {
    // 目标与模块
    "target": "ES2022",
    "module": "ESNext",
    "moduleResolution": "Bundler",
    "moduleDetection": "force", // 新增：确保每个文件都被当成 module
    "lib": ["ES2022", "DOM", "DOM.Iterable"],
    "jsx": "preserve",

    // 严格度（项目铁律）
    "strict": true,
    "noUncheckedIndexedAccess": true,
    "noImplicitOverride": true,
    "noFallthroughCasesInSwitch": true,
    "noUnusedLocals": true, // 新增：未用变量报错（配合 eslint 双保险）
    "noUnusedParameters": true, // 新增
    "noPropertyAccessFromIndexSignature": true, // 新增：防止 obj.foo 意外访问 index signature
    "exactOptionalPropertyTypes": true, // 新增：`prop?: T` 不等同于 `prop: T | undefined`

    // 解析与兼容
    "forceConsistentCasingInFileNames": true,
    "esModuleInterop": true,
    "skipLibCheck": true,
    "isolatedModules": true,
    "resolveJsonModule": true,
    "allowJs": false,
    "verbatimModuleSyntax": true, // 新增：强制 `import type` 语义清晰

    // 输出
    "noEmit": true, // 新增：base 默认不 emit；需要 emit 的子包自行覆盖
    "types": [], // 不自动加载 @types/*，由子包显式声明
  },
  "exclude": ["**/node_modules", "**/dist", "**/.next", "**/build"],
}
```

**审计说明**：

- 现有字段全部保留
- 新增 8 项严格度 / 模块语义 / 构建约定字段
- `noEmit: true` 是 base 默认值 —— 我们所有子包当前都只做 typecheck，不 emit；未来有包需要 emit（比如 `packages/ui` 发 dist）时自行覆盖

### 5.2 各子包 `tsconfig.json`

**`packages/types/tsconfig.json`**（其他 5 个 packages 模板相同，只改路径无关字段）：

```jsonc
{
  "extends": "../config/tsconfig.base.json",
  "compilerOptions": {
    "rootDir": "./src",
    "baseUrl": ".",
  },
  "include": ["src/**/*.ts", "src/**/*.tsx"],
  "exclude": ["node_modules", "dist"],
}
```

**`apps/web/tsconfig.json`**（临时最小版，P0.C-1 会接 Next.js 设置）：

```jsonc
{
  "extends": "../../packages/config/tsconfig.base.json",
  "compilerOptions": {
    "rootDir": "./src",
    "baseUrl": ".",
  },
  "include": ["src/**/*.ts", "src/**/*.tsx"],
  "exclude": ["node_modules", ".next", "dist"],
}
```

**不给 `packages/config` 写 tsconfig.json**：它不含 TS 源码。如果以后 pnpm/IDE 抱怨找不到 tsconfig，届时开新 spec 加一份"空壳" tsconfig，当前不加。

### 5.3 子包 `package.json` 脚本改动

**6 个 `packages/*`**（`types` / `ui` / `prompts` / `agents` / `llm` / `tts`）：

```diff
-   "typecheck": "echo skip",
+   "typecheck": "tsc --noEmit",
```

**`apps/web`**：保持不变（`"typecheck": "echo skip"`），P0.C-1 切换。

**`packages/config`**：保持不变（没有 TS 源码）。

**需要装 TypeScript 作为子包依赖吗？** 不需要。

- 根 `package.json.devDependencies.typescript` 已存在（P0.A-1 装好）
- pnpm workspace 会把根的 `typescript` 二进制 hoist 到每个子包可见的 `node_modules/.bin/tsc`
- 因此子包 `package.json` 里**不加** `typescript` 到它们的 `devDependencies`（保持 P0.A-1 的"TS 集中在根"原则）

如果 hoist 失效（某些 pnpm 配置下），降级方案是在根加 `.npmrc` 的 `public-hoist-pattern[]=typescript`，但默认配置下 pnpm workspace 的 `link-workspace-packages` + 根 devDep 的组合就能让 `tsc` 可见。

### 5.4 `packages/types/src/index.ts` 占位升级

从：

```ts
export {};
```

改为：

```ts
/**
 * @spec docs/specs/P0.A-2-typescript-baseline.md
 * Placeholder type — 让 `pnpm -r typecheck` 有真实代码可检查。
 * P0.B-1 会用 data-model.md 的完整领域类型替换本文件。
 */
export type Placeholder = {
  readonly _brand: "placeholder";
};
```

**为什么要这步**：不做这步的话，`tsc --noEmit` 在"只有 `export {};`"的文件上也能 pass，没法证明我们真的在 typecheck。加一个哪怕最小的类型声明，配合后面的"反向验收"（§ 8 AC-5）证明基线可用。

其他 5 个 packages 继续 `export {};`（它们不是"证明链"的一部分）。

## 7. 失败模式

| 场景                                                            | 行为                                                                                         |
| --------------------------------------------------------------- | -------------------------------------------------------------------------------------------- |
| 子包找不到 `tsc` 二进制                                         | 降级方案：在根 `.npmrc` 加 `public-hoist-pattern[]=typescript`；重装                         |
| `extends` 相对路径写错                                          | `tsc` 报 `File '...' not found`，指出具体路径 —— 修正                                        |
| 某个 package 开启 strict 后旧代码报错                           | 目前所有包都是 `export {};` 或 Placeholder，不会报 —— 如果真报了，说明有未清理 stale，fix 它 |
| `verbatimModuleSyntax` 导致后续引入的 `import { Foo }` 写法报错 | 预期行为：类型引入必须写 `import type { Foo }`；这是项目风格约束                             |

## 8. 验收标准

- **AC-1**: **Given** 所有文件就位，**When** 执行 `pnpm -r typecheck`，**Then** exit 0；6 个 `packages/*`（types / ui / prompts / agents / llm / tts）输出 `tsc --noEmit`；`apps/web` 和 `packages/config` 输出 `echo skip`
- **AC-2**: **Given** base config 到位，**When** 在 `packages/config/tsconfig.base.json` 里把 `strict` 临时改为 `false` 并重跑 typecheck，**Then** 所有子包**仍然** typecheck 通过（因为没真代码），但 `tsc --showConfig` 显示每个包读到的 `strict: false` —— 证明 extends 链路生效。**本 AC 不自动执行，仅作调试用，验证完恢复 strict: true**
- **AC-3**: **Given** `packages/types/src/index.ts` 是 `Placeholder` 定义，**When** 执行 `pnpm --filter @aiclassroom/types typecheck`，**Then** exit 0
- **AC-4**: **Given** `apps/web/tsconfig.json` 存在（但 typecheck 脚本仍是 `echo skip`），**When** 执行 `pnpm --filter web typecheck`，**Then** exit 0，打印 `skip`；P0.C-1 接入 Next.js 后将该脚本切换为 `tsc --noEmit`
- **AC-5**（反向验收）: **Given** 基线就绪，**When** 在 `packages/types/src/index.ts` 临时加一行 `const x: number = "not a number";` 并执行 `pnpm --filter @aiclassroom/types typecheck`，**Then** 命令 exit **非 0** 并报 `TS2322` 类错误。**验证后立即回退这行改动**
- **AC-6**: **Given** 全部 AC 通过，**When** 检查 6 个 `packages/*` 的 `package.json`（不含 config），**Then** `typecheck` 脚本均为 `"tsc --noEmit"`；`apps/web` 和 `packages/config` 仍为 `"echo skip"`

## 9. 开放问题

- **Q1**：`apps/web/tsconfig.json` 现在不包含 Next.js 所需的 `plugin`、`paths`、`incremental`、`allowJs` 等字段。P0.C-1 会直接覆写吗？**预期**：P0.C-1 执行 `create-next-app` 或手动建 Next 项目时会读已有 `tsconfig.json` 并智能合并；若冲突 P0.C-1 里明确覆写且在该 spec 说明。本 spec 不提前处理
- **Q2**：`verbatimModuleSyntax: true` 会不会与 Next.js App Router 的某些 `import` 模式冲突？**风险评估**：主要影响是类型导入必须写 `import type`，与 React 19 / Next 16 默认规则一致。如果 P0.C-1 时发现冲突，优先保留 `verbatimModuleSyntax`，修改用法而非关闭此规则
- **Q3**：`apps/web` 的 web `package.json` 现在没有 `typescript` devDep，也没有 `@types/node` / `@types/react` —— 跑 `tsc --noEmit` 时对一个**空 include**不会报错；但一旦 P0.C-1 往 `src/app/page.tsx` 写代码就会报 `Cannot find type 'React'` 等。是否在 P0.A-2 提前加这些 types？**决定**：不加。P0.A-2 只确保 baseline 生效，web 的类型依赖由 P0.C-1 接入 Next.js 时统一装
- **Q4**：`apps/web` 暂时不跑 `tsc --noEmit`（`echo skip` 占位），是否会让 P0.C-1 忘记切换？**决定**：在 [docs/PLAN.md](../PLAN.md) P0.C-1 任务行的 Artifacts 列显式追加 "把 apps/web.typecheck 从 echo skip 切到 tsc --noEmit"；此外本 spec § 3.1 / § 8 AC-4 已显式记录此欠账
