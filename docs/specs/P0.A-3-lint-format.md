## 1. 背景

P0.A-2 让 TypeScript 的 strict 基线生效，但 TS 的 typecheck **只管类型安全**，不管代码风格、未导入类型、import 顺序、禁用 `any`、Prettier 格式化。这些由 ESLint + Prettier 完成。

本 spec 接入：

- **ESLint 9** 的新版 **flat config** (`eslint.config.mjs`)，根级单文件驱动全仓
- **typescript-eslint v8**（flat config 原生支持）
- **Prettier 3** + `prettier-plugin-tailwindcss`（已在 P0.A-1 装入根 devDep）
- **禁用 `any`** 作为硬规则（[CLAUDE.md § 代码约定](../../CLAUDE.md) 铁律）
- 与 `verbatimModuleSyntax: true`（P0.A-2）**配套**的 `@typescript-eslint/consistent-type-imports: error`，强制 `import type`
- 6 个 `packages/*` 的 `lint` 脚本从 `echo skip` 切到 `eslint .`
- `apps/web` 继续 `echo skip`（同 P0.A-2 的欠账，由 P0.C-1 切换）

**为什么用 flat config 而不是 legacy `.eslintrc`**：

- ESLint 9 已默认 flat config，legacy 配置在 v10 会移除
- 全仓单文件，不用维护 `.eslintrc` 的继承链和 `overrides` 嵌套
- typescript-eslint v8 的 `tseslint.config()` helper 让 ts/js 分流非常简洁

**为什么不用 XO / Biome / Rome 等替代方案**：

- **Biome** 性能更好，但 rule 生态不如 eslint + typescript-eslint 成熟；特别是 `consistent-type-imports` 这种对项目铁律至关重要的规则
- **XO** 偏 opinionated，和我们的手动规则集冲突
- 保持与大多数企业项目一致的选型，降低贡献门槛

如果 Phase 2 后 CI 时间变成瓶颈，可以开新 spec 评估 Biome。

## 2. 用户故事

- 作为开发者，我在任意 `.ts` 文件里写 `const x: any = 1`，`pnpm lint` **必须失败**并指出该行
- 作为开发者，我在 `.tsx` 里写 `import { ComponentType } from "react"`（仅用作类型），lint 报错要求改为 `import type { ComponentType } from "react"`
- 作为贡献者，我保存文件后编辑器自动 format（由 Prettier 驱动），引号、分号、Tailwind class 顺序立刻规范化
- 作为 AI Coding Agent，我生成的代码必须能直接过 `pnpm lint`，否则我应该把它当作错误并修复

## 3. 范围

### 3.1 In Scope

- 根 `eslint.config.mjs`（唯一）、根 `.prettierrc.json`、根 `.prettierignore`
- 根 `devDependencies` 新增：
  - `eslint` (^9.x)
  - `@eslint/js` (^9.x)
  - `typescript-eslint` (^8.x) — 统一包，含 parser + plugin
  - `eslint-config-prettier` (^9.x) — 关闭与 Prettier 冲突的格式化规则
  - `globals` (^15.x) — 环境全局变量定义
- **6 个 `packages/*`**（types / ui / prompts / agents / llm / tts）的 `lint` 脚本：`echo skip` → `eslint .`
- **`apps/web`**：继续 `echo skip`（P0.C-1 切换）
- **`packages/config`**：继续 `echo skip`（无源码）
- 根 `package.json` 新增脚本：
  - `"format": "prettier --write ."`
  - `"format:check": "prettier --check ."`
- 核心 lint 规则集：
  - `@eslint/js` recommended
  - `typescript-eslint` recommended-type-checked
  - `@typescript-eslint/no-explicit-any: error`
  - `@typescript-eslint/consistent-type-imports: error` + `fixStyle: "inline-type-imports"`
  - `@typescript-eslint/no-unused-vars: ["error", { argsIgnorePattern: "^_" }]`
  - `no-console: ["warn", { allow: ["warn", "error"] }]`
  - `eqeqeq: "error"`

### 3.2 Out of Scope

- `eslint-plugin-import` 的 import 顺序规则 — 先不加；如果后续 code review 发现混乱再开 spec
- `eslint-plugin-react` / `eslint-plugin-react-hooks` — 等 P0.C-1 引入 React 时一并接入（届时 spec 会增加 React 规则）
- `eslint-plugin-tailwindcss` — Prettier 的 `prettier-plugin-tailwindcss` 已负责 class 排序；运行时 class 名称检查留给 P0.C-1
- `lint-staged` + git hooks — P0.A-5
- `pnpm check` 聚合命令 — P0.A-5
- 编辑器配置（`.editorconfig` / `.vscode/settings.json`） — 下方 § 5.5 顺手加，不作为主目标

## 5. 关键文件内容

### 5.1 `eslint.config.mjs`

```js
// @ts-check
import js from "@eslint/js";
import tseslint from "typescript-eslint";
import prettier from "eslint-config-prettier";
import globals from "globals";

export default tseslint.config(
  // 1. 忽略
  {
    ignores: [
      "**/node_modules/**",
      "**/dist/**",
      "**/.next/**",
      "**/build/**",
      "**/coverage/**",
      "**/*.config.{js,cjs,mjs}",
      "**/pnpm-lock.yaml",
    ],
  },

  // 2. 基础 JS 规则
  js.configs.recommended,

  // 3. TypeScript 规则（type-checked 需要 tsconfig）
  ...tseslint.configs.recommendedTypeChecked,

  // 4. 项目铁律
  {
    files: ["**/*.{ts,tsx}"],
    languageOptions: {
      parserOptions: {
        projectService: true,
        tsconfigRootDir: import.meta.dirname,
      },
      globals: { ...globals.browser, ...globals.node },
    },
    rules: {
      "@typescript-eslint/no-explicit-any": "error",
      "@typescript-eslint/consistent-type-imports": ["error", { fixStyle: "inline-type-imports" }],
      "@typescript-eslint/no-unused-vars": [
        "error",
        { argsIgnorePattern: "^_", varsIgnorePattern: "^_" },
      ],
      "no-console": ["warn", { allow: ["warn", "error"] }],
      eqeqeq: "error",
    },
  },

  // 5. Prettier 关掉格式化冲突规则（必须放最后）
  prettier,
);
```

**关键点**：

- `projectService: true` — typescript-eslint v8 的新 API，自动从 CWD 向上寻找最近的 `tsconfig.json`，不用手写 `project: [...paths]`
- `tsconfigRootDir` 固定为根，保证 monorepo 里每个子包解析到**自己的** tsconfig
- `prettier` 配置放最后，覆盖所有与格式化相关的规则
- 规则只应用到 `**/*.{ts,tsx}`，JS 文件（比如 config 本身）只走 `js.configs.recommended`

## 7. 失败模式

| 场景                                                                                            | 行为                                                                                                                      |
| ----------------------------------------------------------------------------------------------- | ------------------------------------------------------------------------------------------------------------------------- |
| `eslint .` 在 `packages/config` 等无源码包跑失败                                                | config 继续 `echo skip`，不走 eslint                                                                                      |
| `projectService` 找不到 tsconfig                                                                | 每个子包都有 tsconfig.json（P0.A-2 已建），不会触发                                                                       |
| Prettier 格式化与 ESLint 规则冲突                                                               | `eslint-config-prettier` 关掉所有与格式相关的规则；若仍冲突说明 config 顺序错，把 `prettier` 放最后                       |
| 新装的 ESLint 版本与 typescript-eslint 不兼容                                                   | spec 里锁 caret 范围，install 后看 lockfile；若冲突立即开 spec 修订                                                       |
| Windows 上 `endOfLine: lf` 与 git autocrlf 冲突                                                 | 暂时容忍；未来用 `.gitattributes` 统一                                                                                    |
| `tseslint.configs.recommendedTypeChecked` 对 `packages/*/src/index.ts` 仅 `export {};` 报规则错 | recommended-type-checked 不应该对空文件报错；若报，降级到 `recommended`（非 type-checked）作为 fallback，在 spec Q 里记录 |

## 8. 验收标准

- **AC-1**: **Given** 所有文件就位，**When** 执行 `pnpm -r lint`，**Then** exit 0；6 个 `packages/*` 跑 `eslint .`，其余跑 `echo skip`
- **AC-2**: **Given** 基线就绪，**When** 执行 `pnpm lint`（根聚合），**Then** exit 0
- **AC-3**: **Given** `.prettierrc.json` 就位，**When** 执行 `pnpm format:check`，**Then** exit 0（首次跑前可能需要 `pnpm format` 一次修正既有文件）
- **AC-4**（反向验收 - `any`）: **Given** 基线就绪，**When** 在 `packages/types/src/index.ts` 加一行 `export const bad: any = 1;` 并执行 `pnpm --filter @aiclassroom/types lint`，**Then** exit **非 0**，输出含 `@typescript-eslint/no-explicit-any`。验证后**立即回退**
- **AC-5**（反向验收 - type import）: **Given** 基线就绪，**When** 在 `packages/types/src/index.ts` 加 `import { readFileSync } from "node:fs"; export type T = typeof readFileSync;`（仅作为类型），**Then** lint 报 `@typescript-eslint/consistent-type-imports`，提示应写 `import { type readFileSync }`。验证后**立即回退**
- **AC-6**: **Given** 新 devDep 装完，**When** 检查根 `package.json`，**Then** 包含 `eslint` / `typescript-eslint` / `@eslint/js` / `eslint-config-prettier` / `globals` 五个新增条目
- **AC-7**: **Given** 6 个 `packages/*/package.json`，**When** 检查 `lint` 脚本，**Then** 均为 `"eslint ."`；`apps/web` 和 `packages/config` 仍为 `"echo skip"`

## 9. 开放问题

- **Q1**：`recommendedTypeChecked` 这套规则会对只有 `export {};` 的文件报错吗？**预期**：不会。如果实际踩坑，降级为 `recommended`（非 type-checked）并在 spec 更新记录
- **Q2**：是否在本 spec 顺手加 VSCode `.vscode/extensions.json` 推荐 ESLint + Prettier 插件？**决定**：不加。`.vscode/` 在 P0.A-1 的 `.gitignore` 里被部分排除（只保留 `settings.json` / `extensions.json`）；但真要写就会进入 IDE 集成领域，另开小 spec 更干净
- **Q3**：`no-console` 用 `warn` 还是 `error`？**决定**：`warn`。开发阶段允许临时 `console.log`，但 `pnpm check` 不会因 warn 失败；生产代码走 `pino`（tech-stack.md § 3 已定）
- **Q4**：`prettier-plugin-tailwindcss` 在当前没有 Tailwind 文件时会报错吗？**预期**：不会。插件只在遇到 JSX/HTML `className` / `class` 时才排序；对纯 `.ts` 文件无副作用
