## 1. 背景

P0.A-3 有了 lint + format:check，P0.A-4 有了 test，P0.A-2 有了 typecheck。四条独立流水线散落在四条命令里，谁也不保证"提交前全跑一遍"。

本 spec 目标：

1. **`pnpm check`** 作为**单一入口**，一条命令串起 `format:check + lint + typecheck + test`，失败任意一条即 exit 非 0 —— 给人类与 AI Coding Agent 一个"上船前的统一闸门"
2. **simple-git-hooks + lint-staged** 接入 **pre-commit hook**，在本地 commit 时**只对 staged 文件**跑 `eslint --fix + prettier --write`，把"提交前自动整理"的成本降到零
3. 明确 **pre-push** 不做重活（不挂 full check），原因见 § 3.2

**为什么 `pnpm check` 要包含 `format:check` 而不只是 `lint + typecheck + test`**：当前 `package.json` 的 `check` 脚本漏了 format:check，意味着"检查"通不了的格式错误只会在 code review 时被发现。一次修到位。

**为什么选 simple-git-hooks 而不是 husky**：

- tech-stack.md § 131 已锁定
- simple-git-hooks **零运行时开销**：就是一个把 `package.json` 里的配置写进 `.git/hooks/*` 的 postinstall 脚本；husky 会往仓库里塞一个 `.husky/` 目录并把 `core.hooksPath` 改掉
- 配置收敛到 `package.json` 一处，与 monorepo 风格一致
- 代价：没有 husky 的分层 hooks 管理；但我们只用 1-2 个 hook，用不上

**为什么 pre-commit 只跑 lint-staged 不跑 full check**：

- `pnpm check` 完整跑包括 `pnpm -r test`，本地开发一次提交等几十秒 → 会逼人 `--no-verify`
- lint-staged 只扫 staged 文件，大部分提交 <1s 完成
- 真正的全量检查由 CI（Phase 0 不引入，留给 Phase 4）或开发者手动 `pnpm check` 保证

## 2. 用户故事

- 作为开发者，我提交前跑一条 `pnpm check` 就知道自己的改动是否"船上干净" —— 格式、lint、类型、测试四条线一次通过
- 作为开发者，我 `git commit` 时不用手动跑 prettier —— pre-commit hook 会自动 `prettier --write` + `eslint --fix` 我 staged 的文件，修完自动 `git add` 回去
- 作为 AI Coding Agent，我完成一个 spec 的实现后**必须**跑一次 `pnpm check`，全绿才能声称"完成"
- 作为 code reviewer，我看到 PR 时可以假设"格式 + lint + typecheck 一定过了"，评审能聚焦在逻辑和设计上

## 3. 范围

### 3.1 In Scope

- 修改根 `package.json` 的 `"check"` 脚本：`lint && typecheck && test` → `format:check && lint && typecheck && test`
- 新增根 `devDependencies`：
  - `simple-git-hooks` (^2.x)
  - `lint-staged` (^15.x)
- 根 `package.json` 顶层新增两个配置块：
  - `"simple-git-hooks": { "pre-commit": "pnpm lint-staged" }`
  - `"lint-staged": { ... }`（glob 映射到 eslint --fix / prettier --write，见 § 5.3）
- 根 `package.json` 新增 `postinstall` 脚本，让 `pnpm install` 自动 `simple-git-hooks`（幂等）
- `git init`（如果 Q1 决定包含）
- 反向验证 hook：故意提交一个 `any` → pre-commit 失败

### 3.2 Out of Scope

- **pre-push hook**：不加。理由：
  - pre-push 要跑 full `pnpm check` 才有价值，几十秒延迟会逼人 `--no-verify`
  - push 失败的保护应该在 CI，不是本地 hook
  - 将来 CI spec 会补上 push → GitHub Actions → full check
- **commit-msg hook**（commitlint / conventional commits）：不加。提交信息规范留给后续 spec，当前 Phase 0 没必要卡这层
- **husky**：明确不用（见 § 1）
- **CI 流水线**（GitHub Actions / GitLab CI）：留到 Phase 4 企业级
- **coverage 门槛 / 增量 coverage**：留到独立 spec；本 spec 不动 coverage
- **lint-staged 的 `*.md` 处理**：只对 `*.{ts,tsx,js,mjs,cjs,json,md,mdx,css}` 跑 prettier，**不对 markdown 跑 eslint**；`*.md` 不需要 lint
- **根 CI 脚本 `pnpm ci`**：P0.A-5 不需要，`pnpm check` 已经能用

### 3.3 Open — 由 Q1 决定

- **是否在本 spec 内 `git init`**：如果不做，hook 配置文件会就位但**实际 hook 不会安装**，AC-6 反向验证就跑不了（没有 `.git/hooks/pre-commit`）。决策见 § 9 Q1

## 5. 关键文件内容

### 5.1 根 `package.json` 完整 diff

```diff
  "scripts": {
    "dev": "pnpm --filter web dev",
    "build": "pnpm -r build",
    "lint": "pnpm -r lint",
    "format": "prettier --write .",
    "format:check": "prettier --check .",
    "typecheck": "pnpm -r typecheck",
    "test": "pnpm -r test",
-   "check": "pnpm lint && pnpm typecheck && pnpm test"
+   "check": "pnpm format:check && pnpm lint && pnpm typecheck && pnpm test",
+   "postinstall": "simple-git-hooks || echo \"[simple-git-hooks] skip (no .git)\""
  },
  "devDependencies": {
    "@eslint/js": "^9.15.0",
    "@vitest/coverage-v8": "^3.0.0",
    "eslint": "^9.15.0",
    "eslint-config-prettier": "^9.1.0",
    "globals": "^15.12.0",
+   "lint-staged": "^15.2.10",
    "prettier": "^3.3.3",
    "prettier-plugin-tailwindcss": "^0.6.8",
+   "simple-git-hooks": "^2.11.1",
    "typescript": "^5.6.0",
    "typescript-eslint": "^8.15.0",
    "vitest": "^3.0.0"
  },
+ "simple-git-hooks": {
+   "pre-commit": "pnpm lint-staged"
+ },
+ "lint-staged": {
+   "*.{ts,tsx}": [
+     "eslint --fix",
+     "prettier --write"
+   ],
+   "*.{js,mjs,cjs,json,md,mdx,css,yml,yaml}": [
+     "prettier --write"
+   ]
+ }
```

**关键点**：

- `"check"` 顺序：**format:check 第一**。理由：格式化最快、最纯粹，秒级返回；让它做"快速失败"的第一道闸门，避免跑完 30 秒 test 才告诉你 tab 错了
- `postinstall` 容错 `|| echo ...`：当 `.git` 不存在时 `simple-git-hooks` 会报错，用 `|| echo` 吞掉让 `pnpm install` 不中断。Q1 决定 git init 后这条容错保留（代价为零，好处是未来 clone 且 `--filter` 跳过 hooks 时也不会炸）
- `lint-staged` 对 `*.{ts,tsx}` 先 `eslint --fix` 再 `prettier --write`：eslint 可能改格式（比如 `consistent-type-imports` 的 fixer 会插 `type`），然后 prettier 把它们格式化对齐。顺序反了会留下 eslint 未来一跑就 diff 的 stale 状态
- **不**对 `*.md` 跑 eslint（markdown 不在 eslint flat config 的 `files` 范围内，跑了也是 no-op + 噪音）
- 所有 JSON / YAML 都走 prettier，保证 `package.json` 里的 JSON 格式也被自动对齐

### 5.2 `simple-git-hooks` 工作机制 recap

`simple-git-hooks` 的 postinstall 脚本读取 `package.json` 里的 `"simple-git-hooks"` 对象，把每条键值写进 `.git/hooks/<hook-name>`，并 `chmod +x`。**不需要额外的目录或 runner**。

`pnpm lint-staged` 会从 `package.json` 里读 `"lint-staged"` 对象，对 staged 文件按 glob 分组跑命令；每条命令的退出码决定 hook 是否通过。

## 6. 实现步骤

```
1. 改根 package.json：
   - check 脚本加 format:check
   - 新增 postinstall
   - 新增 simple-git-hooks / lint-staged 两个顶层 config 块
   - 新增 2 个 devDep
2. pnpm install（拉 2 个新 devDep；若 .git 存在，postinstall 自动装 hook）
3. Q1 分叉：
   3a. 若做 git init → git init + git add + git commit → 再跑一次 pnpm install 触发 postinstall 装 hook
   3b. 若不做 → 跳到 4
4. 跑 pnpm check 验证 AC-1（全绿）
5. AC-3 反向验收（check 任一环失败则整条失败）
6. 若 3a 已做 → 跑 AC-5 / AC-6（hook 真实拦截）；否则 AC-5 / AC-6 标 deferred
7. 置 implemented + 勾 PLAN.md
```

## 7. 失败模式

| 场景                                                                 | 行为                                                                                                                             |
| -------------------------------------------------------------------- | -------------------------------------------------------------------------------------------------------------------------------- |
| `.git` 不存在时 `pnpm install` 触发 `simple-git-hooks` 报错          | `postinstall` 脚本末尾的 `\|\| echo ...` 兜底吞掉错误码，不中断 install                                                          |
| Windows 下 `simple-git-hooks` 写入的 hook 文件没有可执行位           | Git for Windows 会忽略 Unix 可执行位，按脚本内容解释，一般能跑；若报错参考 simple-git-hooks README Windows 节                    |
| `lint-staged` 对已 `git add` 的文件自动 fix 后忘记 re-add            | lint-staged 默认行为就是 fix 后自动 `git add`，不需要手动处理                                                                    |
| `eslint --fix` 改了文件但 `prettier --write` 又反悔                  | `eslint-config-prettier` 已关闭所有与格式相关的 eslint 规则，两者应该互补；若仍冲突说明某条规则需要补进 `eslint-config-prettier` |
| pre-commit hook 在 CI 上也跑                                         | CI 用 `git commit -n` 或不走 commit 路径；simple-git-hooks 只影响本地 .git/hooks                                                 |
| `pnpm check` 在中文 Windows 终端乱码                                 | 已在 P0.A-4 § 7 讨论过；非 spec 关注点                                                                                           |
| pre-commit 自动 fix 后，文件与开发者预期不符（比如 import 顺序被改） | 这是 lint-staged 本身的权衡；用户可以 `git commit --no-verify` 临时绕过；长期靠教育                                              |
| `pnpm -r test` 偶发网络/端口冲突导致 `pnpm check` 假阳失败           | 当前 vitest 没用网络，不适用；未来 e2e 测试进来时另开 spec 改 check 策略                                                         |

## 8. 验收标准

- **AC-1**: **Given** 所有配置就位，**When** 执行 `pnpm check`，**Then** exit 0；输出包含 `format:check` / `lint` / `typecheck` / `test` 四段
- **AC-2**: **Given** 基线就绪，**When** 检查根 `package.json`，**Then** 包含 `"simple-git-hooks"` 和 `"lint-staged"` 两个顶层配置块，`"check"` 脚本首段是 `pnpm format:check`
- **AC-3**（反向 - check 快速失败）: **Given** 在任一 `*.ts` 文件里制造一个 lint 错误（如 `const x: any = 1`），**When** 执行 `pnpm check`，**Then** exit **非 0**，且在 `lint` 段失败，不继续跑 `typecheck` / `test`。验证后**立即回退**
- **AC-4**（反向 - check 格式优先）: **Given** 在任一 `*.ts` 文件里制造一个 prettier 能发现的格式错误（如把 `const x = 1;` 改成 `const x   =   1;`），**When** 执行 `pnpm check`，**Then** exit **非 0**，且在 **`format:check`** 段就失败（证明顺序正确）。验证后**立即回退**
- **AC-5**（hook 真实拦截，依赖 Q1=是）: **Given** git repo 就绪，`.git/hooks/pre-commit` 已安装，**When** 在 `packages/tts/src/index.ts` 加 `const bad: any = 1;` 并 `git add . && git commit -m "test"`，**Then** commit 被 pre-commit **拦截**（exit 非 0，commit 未产生），输出含 `eslint --fix` 报错。验证后**立即回退**
- **AC-6**（hook 自动 fix 后成功，依赖 Q1=是）: **Given** git repo 就绪，**When** 在 `packages/tts/src/index.ts` 加一行 `const x=1` (缺空格 + 缺分号) 并 `git add . && git commit -m "test2"`，**Then** pre-commit 自动 fix 为 `const x = 1;`，commit 成功。验证后**立即回退**
- **AC-7**（不退化）: **Given** 本 spec 完成，**When** 执行 `pnpm check`，**Then** exit 0（再次全绿，防 P0.A-2/3/4 被污染）

## 9. 开放问题

- **Q1**（最关键）：**本 spec 是否 `git init` 并做首次 commit？**
  - **选项 A**：做。优点：AC-5/6 能真实跑；未来所有变更进版本管理；符合企业级项目习惯。缺点：第一个 commit 巨大（含 `pnpm-lock.yaml` 和所有 Phase 0.A 产物），且 `git init` 是一次性的单向动作
  - **选项 B**：不做。优点：保持本 spec 纯工具链；git init 作为独立仪式留给用户。缺点：AC-5/6 跑不了，hook 配置是"死的"直到用户 git init
  - **选项 C**：本 spec 做配置，AC-5/6 标 deferred，新开一条 `P0.A-6-git-init.md`（超轻 spec）专门做 git init 并把 AC-5/6 搬过去
  - **推荐**：**A**。理由：① `git init` 是 Phase 0 基础设施的自然组成 ② 不做的话这个 spec 有 hook 配置但无法验证，留一个"半完成"状态最糟 ③ git init 本身不是争议动作，拆 spec 过度。等用户拍板
- **Q2**：`"check"` 脚本里 `format:check` 放第一还是 `lint` 放第一？**决定**：`format:check` 第一。格式化最快且纯粹，秒级 fail-fast 避免 30s test 跑完才知道 tab 错了
- **Q3**：要不要把 `pnpm check` 也加到 postinstall / pre-push？**决定**：都不加。postinstall 已经要跑 simple-git-hooks，再加 check 会让 `pnpm install` 变数十秒；pre-push 理由见 § 3.2
- **Q4**：lint-staged 对 `package.json` 的 prettier 会不会改掉我们精心维护的 devDep 顺序？**预期**：不会 —— prettier 对 JSON 只做缩进 / 尾逗号 / 换行，不会重排 key。验证一次确认
- **Q5**：`postinstall` 的容错用 `|| echo ...` 还是用 `node -e "..."` 检查 `.git` 存在再执行？**决定**：`|| echo`。跨平台、无依赖、语义够清楚；代价是退出码被吞掉（故意的）
- **Q6**：lint-staged 里对 `*.{ts,tsx}` 是否同时跑 vitest 相关 test？**决定**：不跑。test 属于 `pnpm check` 的职责；lint-staged 只做"快速 fix"。跑 test 会让 commit 秒变十几秒
