## 1. 背景

P0.A-2 有了 TypeScript 严格基线，P0.A-3 有了 lint / format，但**代码行为正确性**没有自动化防线。`pnpm -r test` 当前全是 `echo skip`，P0.A-5 的 `pnpm check` 如果现在接上就是一条假流水线。

本 spec 目标：把 **Vitest 3.x** 作为项目唯一单元测试框架接入，让 6 个 `packages/*` 的 `test` 脚本从 `echo skip` 切到 `vitest run`，每个包至少有**一条真实断言**，且能被反向验证（故意写错 → `pnpm test` 失败）。

**为什么选 Vitest 而不是 Jest / Node built-in test runner**：

- tech-stack.md § 测试 已锁定 Vitest，此处是落地
- Vite 生态天然兼容 Next.js 16 / React 19 / ESM / TypeScript，不需要 Babel/ts-jest 这层转译
- `tseslint.configs.recommendedTypeChecked` + `vitest` 的 projectService 兼容得比 Jest 好
- Node built-in runner 缺 watch / ui / coverage 的集成体验

**为什么 Testing Library 不在本 spec**：

- 当前仓库里**还没有 React 组件**（apps/web 的 `src/` 是空目录，P0.C-1 才 bootstrap Next.js）
- Testing Library + jsdom 需要一个真实 React 组件做 smoke test，否则装上就是死代码
- 延后到 **P0.C-1** 的 spec 里顺手加（装 `@testing-library/react` + `@testing-library/jest-dom` + `jsdom` + 一条 `Button` 渲染测试），用一条 spec 管两个相邻的事

## 2. 用户故事

- 作为开发者，我在 `packages/types/src/foo.ts` 写新函数时，可以在同目录建 `foo.test.ts`，`pnpm --filter @aiclassroom/types test` 能发现并运行它
- 作为开发者，我在任何包里故意把 `expect(1 + 1).toBe(2)` 改成 `.toBe(3)`，`pnpm -r test` **必须失败**并指出对应文件与行号
- 作为 AI Coding Agent，我修改业务代码后应该跑 `pnpm test` 验证，如果失败我应该修代码而不是改测试
- 作为 P0.A-5 的前置条件，我提供一条真实的 `pnpm -r test` 让 `pnpm check` 能把它接进流水线

## 3. 范围

### 3.1 In Scope

- 根 `vitest.config.ts`（唯一），最小配置：
  - `environment: "node"`（DOM 环境由 P0.C-1 另建）
  - `globals: false`（显式 `import { describe, it, expect } from "vitest"`，与 `verbatimModuleSyntax` 配套）
  - `include: ["src/**/*.test.ts", "src/**/*.test.tsx"]` —— **cwd 相对**：子包跑 `vitest run` 时匹配到自己的 `src/`
  - `passWithNoTests: false`（空包必须至少一条 smoke test，防止"假全绿"）
- 根 `devDependencies` 新增：
  - `vitest` (^3.x)
  - `@vitest/coverage-v8` (^3.x) — 不接 coverage pipeline，仅安装保证 `vitest --coverage` 将来能跑
- **6 个 `packages/*`**（types / ui / prompts / agents / llm / tts）：
  - `test` 脚本：`echo skip` → `vitest run`
  - 新增 **一条** `src/*.test.ts`：对该包 `src/index.ts` 的导出做一次 smoke 断言
- **`apps/web`**：继续 `echo skip`（同 P0.A-2、P0.A-3 欠账，由 P0.C-1 切到带 jsdom 的 Vitest + Testing Library）
- **`packages/config`**：继续 `echo skip`（无源码）
- 根 `package.json` 脚本：**不新增**；保持 `"test": "pnpm -r test"`（已存在）
- ESLint：**不新增**规则（vitest 的 `eslint-plugin-vitest` 留给后续 spec）

### 3.2 Out of Scope

- **Testing Library + jsdom + React 组件测试** — 延后到 P0.C-1 统一处理
- **Coverage 门槛 / CI 报告** — 只装包，不定阈值，不定报告位置
- **`@vitest/ui`** — 交互式 runner UI，非必需
- **Vitest Workspaces (`vitest.workspace.ts`)** — 当前 6 个包共享同一份根 config 就够用，不需要每个包独立 project
- **Mock / MSW / fixture 约定** — 等第一个真正需要 mock 的业务 spec 触发
- **Watch 模式脚本** — 开发者直接 `pnpm vitest` 即可，不在 package.json 写快捷方式
- **快照测试 (`toMatchSnapshot`)** — 约定不使用快照测试（维护成本 > 收益），业务 spec 一律用显式断言

## 5. 关键文件内容

### 5.1 `vitest.config.ts`

```ts
import { defineConfig } from "vitest/config";

export default defineConfig({
  test: {
    environment: "node",
    globals: false,
    include: ["src/**/*.test.ts", "src/**/*.test.tsx"],
    passWithNoTests: false,
    clearMocks: true,
    restoreMocks: true,
  },
});
```

**关键点**：

- `globals: false` — 强制显式 `import { describe, it, expect } from "vitest"`；与 P0.A-2 的 `verbatimModuleSyntax: true` 配合，避免"神秘全局"
- `include` 写成 cwd 相对的 `src/**/*.test.ts` + `src/**/*.test.tsx`：vitest 子进程默认 `root = cwd`，`include` 相对 `root` 解析。`pnpm -r test` 会让每个包从自己的 cwd 跑 `vitest run`，vitest 向上找到根 config、再用自身 cwd 作为 `root`，正好匹配该包 `src/` 下的测试。写成绝对路径 `packages/<name>/src/**` 反而会在子包 cwd 下被解析成 `packages/<pkg>/packages/<name>/src/**`，匹配不到任何文件（这是 Vitest glob 的真实行为，初稿的解释写反了）
- 代价：在**仓库根**直接跑 `vitest run` 不会发现任何测试（根目录没有 `src/`）；这是可以接受的，因为本 spec 不支持"根 cwd 跑 vitest"；需要一口气跑全部走 `pnpm -r test`
- `passWithNoTests: false` — 空包报错，强迫每个包至少一条 smoke test
- `clearMocks` / `restoreMocks` — 测试间 mock 隔离，为未来真·mock 做准备
- **不**启用 coverage，`@vitest/coverage-v8` 装上只是为了 `vitest --coverage` 将来直接能跑

### 5.3 子包 `package.json` 脚本（6 个）

```diff
-   "test": "echo skip",
+   "test": "vitest run",
```

**`apps/web` 和 `packages/config`**：保持 `"test": "echo skip"`。

### 5.4 示例测试内容（6 个包共用模板）

以 `packages/types/src/index.test.ts` 为例：

```ts
import { describe, expect, it } from "vitest";

import type { Placeholder } from "./index";

describe("@aiclassroom/types — placeholder", () => {
  it("Placeholder type can be brand-narrowed", () => {
    const value = { _brand: "placeholder" } as const satisfies Placeholder;
    expect(value._brand).toBe("placeholder");
  });
});
```

**其他 5 个包**的 `src/index.test.ts` 按同样套路：

```ts
import { describe, expect, it } from "vitest";

describe("@aiclassroom/<pkg> — smoke", () => {
  it("1 + 1 = 2", () => {
    expect(1 + 1).toBe(2);
  });
});
```

**为什么不统一"1+1"**：`packages/types` 有真实导出 (`Placeholder`)，顺手对它做一次 `satisfies` 检查能同时测到 TS 类型 + 运行时行为；其他 5 个包 `src/index.ts` 是空占位，写 `1+1` 是目前能做到的最低有效断言，P0.B-\* 系列 spec 落地这些包时会替换为真实测试。

## 6. 实现步骤

```

   a. 改 package.json test 脚本：echo skip → vitest run

```

## 7. 失败模式

| 场景                                                                 | 行为                                                                                                                                                 |
| -------------------------------------------------------------------- | ---------------------------------------------------------------------------------------------------------------------------------------------------- |
| `vitest run` 在某个包找不到测试                                      | `passWithNoTests: false` 会报非 0；意味着该包的 `*.test.ts` 没建，补建后再跑                                                                         |
| 子包 cwd 下 vitest 无法找到根 config                                 | Vitest 默认向上搜索 `vitest.config.*`，会找到根；如果仍失败，说明 Vitest 版本改了行为 —— 回退到给每个包建 1 行的 `vitest.config.ts` extend 根 config |
| 子包跑 `vitest run` 报 "No test files found"                         | `include` 必须是 cwd 相对的 `src/**/*.test.{ts,tsx}` —— 已在初稿修订；若仍踩坑检查根 config 是否被覆盖                                               |
| `@aiclassroom/types` 的 `satisfies` 报错                             | 说明 `Placeholder` 定义变了；优先改测试匹配类型，不改类型去迁就测试                                                                                  |
| `eslint-config-prettier` 与 vitest 断言风格冲突                      | Prettier 不管断言逻辑，只管格式；不会冲突                                                                                                            |
| ESLint 的 `@typescript-eslint/no-unused-vars` 对 `describe/it` 误报  | 这些是函数调用而非变量声明，不会触发；如触发说明 import 写歪了                                                                                       |
| `tseslint.configs.recommendedTypeChecked` 对 `*.test.ts` 文件报慢/错 | test 文件已被 P0.A-2 的 `include: ["src/**/*.ts", "src/**/*.tsx"]` 纳入各包 tsconfig，projectService 能处理；若踩坑，Q1 登记                         |
| 本地 Windows 下 `vitest run` 乱码                                    | Vitest 3 默认 chalk 颜色在 Windows 终端正常，若乱码设 `FORCE_COLOR=0` 或用 Windows Terminal                                                          |

## 8. 验收标准

- **AC-1**: **Given** 所有文件就位，**When** 执行 `pnpm -r test`，**Then** exit 0；6 个 `packages/*` 跑 `vitest run` 并报告 1 passed，其余跑 `echo skip`
- **AC-2**: **Given** 基线就绪，**When** 执行 `pnpm test`（根聚合），**Then** exit 0
- **AC-3**（反向验收）: **Given** 基线就绪，**When** 把 `packages/llm/src/index.test.ts` 的 `expect(1 + 1).toBe(2)` 改成 `.toBe(3)` 并跑 `pnpm --filter @aiclassroom/llm test`，**Then** exit **非 0**，报告含 `expected 2 to be 3`。验证后**立即回退**
- **AC-4**（反向验收 - `passWithNoTests`）: **Given** 基线就绪，**When** 临时删掉 `packages/tts/src/index.test.ts` 并跑 `pnpm --filter @aiclassroom/tts test`，**Then** exit **非 0**，输出含 `No test files found`。验证后**立即恢复文件**
- **AC-5**: **Given** 新 devDep 装完，**When** 检查根 `package.json`，**Then** 包含 `vitest` 与 `@vitest/coverage-v8` 两个新增条目
- **AC-6**: **Given** 6 个 `packages/*/package.json`，**When** 检查 `test` 脚本，**Then** 均为 `"vitest run"`；`apps/web` 和 `packages/config` 仍为 `"echo skip"`
- **AC-7**（不退化）: **Given** 本 spec 完成，**When** 执行 `pnpm lint && pnpm format:check && pnpm typecheck`，**Then** 三条全部 exit 0（P0.A-2 / P0.A-3 的线不被 vitest 文件污染）

## 9. 开放问题

- **Q1**：`tseslint.configs.recommendedTypeChecked` 对 `*.test.ts` 文件会不会因 `it/describe` 回调的 `any` 推断报警？**预期**：不会，Vitest 的类型声明把回调签名声明为 `() => void | Promise<void>`。若踩坑，给测试文件专门加一层 override（放宽 `no-floating-promises`），在本节追更
- **Q2**：要不要在 `packages/types/src/index.test.ts` 用 `satisfies` 还是统一都写 `1+1`？**决定**：types 包用 `satisfies` 展示"类型 + 运行时同时测"的风格，其他 5 个包 `src/index.ts` 尚无导出，写 `1+1` 即可；P0.B-\* 落地时替换
- **Q3**：`@vitest/coverage-v8` 装了但不用，是否浪费？**决定**：不浪费 —— 装上是为了让任何开发者临时跑 `pnpm vitest --coverage` 不用再 pnpm install，是一种"低成本让未来路径畅通"的取舍；真正的 coverage 门槛留给独立 spec
- **Q4**：`vitest.config.ts` 放根还是放 `packages/config/`？**决定**：放根。`packages/config` 在 P0.A-2 只放 TS config，让测试配置跟着"项目级工具链"约定放根；如果将来多 app 需要不同测试配置，再引入 `vitest.workspace.ts`
