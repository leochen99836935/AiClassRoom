## 1. 背景与动机

`docs/data-model.md` 定义了 7 大类核心领域类型（Lesson / Role / Scene / Slide / Action / Trace / Playback）。目前 `packages/types/src/index.ts` 只有一个 `Placeholder` 占位。

本 spec 的目标：**把 data-model.md 中所有 interface 和 type 1:1 落到 TypeScript 文件中**，形成后续所有包（`packages/llm`、`packages/agents`、`packages/prompts`）和应用（`apps/web`）的类型地基。

这条任务风险标"中"，因为：

- 7 种 Action 类型是 **tagged union**（`type` 字段做 discriminant），分错会导致播放引擎类型推断全歪
- Lesson → Scene → Action 的嵌套关系、不变量约束需要在类型层面尽可能表达
- 后续 `packages/llm`（P0.B-3）的 `generateStructured` 需要对这些类型做 zod schema 校验，类型设计要与 zod 友好

## 2. 用户故事

- 作为开发者，我 `import type { Lesson, Scene, Action } from "@aiclassroom/types"`，就能拿到与 `data-model.md` 完全一致的类型定义
- 作为 AI Agent，我在实现 scene_agent / action_agent 时，TS 编译器能告诉我哪些字段必填、Action 的 `type` discriminant 是什么
- 作为 reviewer，我对照 `data-model.md` 能逐行验证 TS 文件，不需要猜

## 3. 范围

### 3.1 In Scope

- 在 `packages/types/src/` 下新建以下文件，每个文件对应 `data-model.md` 的一个章节：
  - `common.ts` — 共享工具类型（`Id`、`ISODateString` 等）
  - `role.ts` — `Role`、`VoiceConfig`
  - `slide.ts` — `Slide`、`SlideElement`、`Rect`、`TextStyle`
  - `whiteboard.ts` — `Whiteboard`、`Stroke`
  - `action.ts` — `ActionBase`、7 种具体 Action、`Action` union、`ActionType`
  - `scene.ts` — `Scene`
  - `lesson.ts` — `Lesson`、`LessonStatus`、`LessonMeta`、`Material`
  - `trace.ts` — `TraceRecord`
  - `playback.ts` — `PlaybackState`、`PlaybackError`
- 重写 `index.ts` 为所有文件的 re-export hub
- 重写 `index.test.ts`，对核心类型做编译时（`satisfies`）和运行时（结构断言）校验
- 删除旧的 `Placeholder` 类型

### 3.2 Out of Scope

- **Zod schema**：本 spec 只做 TS 类型。Zod schema 属于 `packages/llm` 或独立 spec（P0.B-3 或专门的 schema spec）
- **运行时构造函数 / factory / builder**：不在类型包里放业务逻辑
- **ID 生成（nanoid）**：类型层只声明 `id: string`，运行时生成逻辑留给业务代码
- **i18n 类型**：`Language` 类型只在 Lesson 上做 union，不引入 next-intl 类型
- **扩展 data-model.md**：本 spec 严格 1:1 映射，不增不减；如发现缺漏先改 data-model.md

## 5. 数据与 API

### 5.1 涉及的数据类型

全部来自 `docs/data-model.md`，1:1 映射。以下列出各文件导出的类型（不重复贴 interface 定义，以 data-model.md 为准）：

**`common.ts`**

```ts
/** 语义别名，运行时仍是 string，用于文档自描述 */
export type Id = string;
export type ISODateString = string;
```

> 注：`Id` 和 `ISODateString` 是语义 alias，不引入 branded type（data-model.md § 1 约定 ID 为 `string`，不加 brand）。

**`role.ts`** — `VoiceConfig`, `Role`

**`slide.ts`** — `TextStyle`, `Rect`, `SlideElement`（tagged union on `kind`）, `Slide`

> `TextStyle` 在 data-model.md 中被 `SlideElement` 的 `text` kind 引用但未展开定义。本 spec 约定暂定为：
>
> ```ts
> export interface TextStyle {
>   fontSize?: number;
>   fontWeight?: "normal" | "bold";
>   color?: string;
>   align?: "left" | "center" | "right";
> }
> ```
>
> 如后续 Slide 渲染 spec 需要扩展，先改 data-model.md。

**`whiteboard.ts`** — `Stroke`, `Whiteboard`

**`action.ts`** — `ActionBase`, `SpeechAction`, `SlideAction`, `FocusAction`, `WriteAction`, `InteractAction`, `DiscussAction`, `WaitAction`, `Action`（union）, `ActionType`

关键设计决策：

1. **Tagged union discriminant 是 `type` 字段**，值为 `"SPEECH" | "SLIDE" | "FOCUS" | "WRITE" | "INTERACT" | "DISCUSS" | "WAIT"`
2. 每种具体 Action **用 `interface ... extends ActionBase`**，覆写 `type` 字段为字面量
3. `Action = SpeechAction | SlideAction | ...` 是 union type
4. `ActionType = Action["type"]` 从 union 自动提取

**`scene.ts`** — `Scene`（`status: "pending" | "generating" | "ready" | "failed"`）

**`lesson.ts`** — `LessonStatus`, `LessonMeta`, `Material`, `Lesson`

**`trace.ts`** — `TraceRecord`（`input: unknown; output: unknown`，符合 strict 禁 any 约定）

**`playback.ts`** — `PlaybackError`, `PlaybackState`

## 7. 编码约定（补充 CLAUDE.md 通用规则）

- 每个文件头部加 `/** @spec docs/specs/P0.B-1-domain-types.md */`
- 所有 interface/type 加 **JSDoc 一行描述**，不堆砌注释
- 字段名 camelCase，类型名 PascalCase（data-model.md § 1）
- 枚举用 union string type，不用 TS `enum`（data-model.md § 1）
- 可选字段显式标 `?`（data-model.md § 1）
- `import type` 用 type-only import（`consistent-type-imports` 规则）
- 文件之间的 import 关系：`lesson → scene, role, material` / `scene → slide, whiteboard, action` / `action → whiteboard`（Stroke）

## 8. 失败模式

| 场景                                                 | 行为                                                           |
| ---------------------------------------------------- | -------------------------------------------------------------- |
| data-model.md 与本 spec 描述不一致                   | 以 data-model.md 为准，先改本 spec 再改代码                    |
| `TextStyle` 需要新增字段                             | 先改 data-model.md § 2.4，再改 slide.ts                        |
| 某个 Action 类型的 `extends ActionBase` 导致多余字段 | 使用 `interface extends` + 显式覆写 `type` 字段，TS 会严格窄化 |
| 循环 import                                          | 设计上不存在：action 不 import scene，scene 不 import lesson   |

## 9. 验收标准 (Acceptance Criteria)

- **AC-1**: **Given** 所有类型文件就位，**When** 执行 `pnpm check`，**Then** exit 0（lint + typecheck + test 全绿）
- **AC-2**: **Given** `packages/types/src/index.ts` 被 re-export，**When** 在任意包中 `import type { Lesson, Scene, Action, Role, Slide, TraceRecord, PlaybackState } from "@aiclassroom/types"`，**Then** 所有类型可用且与 data-model.md 一致
- **AC-3**: **Given** Action union，**When** 对 `Action` 变量做 `switch (action.type)` exhaustive check，**Then** TS 在 default 分支推断 `action` 为 `never`（证明 tagged union 正确）
- **AC-4**: **Given** `data-model.md` § 5 不变量，**When** 构造一个 `FocusAction` 且不填 `elementId`，**Then** TS 编译报错
- **AC-5**: **Given** `index.test.ts` 中用 `satisfies` 构造每种 Action 的最小合法值，**When** 执行 `pnpm -r test --filter @aiclassroom/types`，**Then** 全部通过
- **AC-6**: **Given** 本 spec 完成，**When** 逐一对比 `data-model.md` §2-§4 中每个 `interface` / `type`，**Then** `packages/types/src/` 中有完全对应的导出（不多不少）

## 10. 测试计划

`packages/types/src/index.test.ts` 覆盖：

1. **编译时校验**：用 `satisfies` 为每种核心类型构造一个最小合法对象，确保类型签名正确
2. **Action exhaustive switch**：写一个 `function getActionLabel(a: Action): string`，在 default 分支 `const _: never = a`，证明 union 覆盖完整
3. **结构断言**：`expect(typeof minimalLesson.id).toBe("string")` 等基础运行时断言，确保 `satisfies` 不只是 type-level 空转

不需要：

- 组件测试（无 UI）
- E2E（无交互）
- Zod 校验测试（Zod 不在本 spec 范围）

## 13. 开放问题

- **Q1**: `TextStyle` 在 data-model.md 中未展开定义。本 spec 暂定 `{ fontSize?, fontWeight?, color?, align? }`。需要用户确认是否合理，或是否应先补充 data-model.md。
- **Q2**: `SlideElement` 的 `shape` kind 是否需要 `style` 字段（颜色、填充等）？当前 data-model.md 没有，本 spec 不加。
- **Q3**: `InteractAction.onAnswer` 中的 `branchToActionId` 引用的是同 Scene 的 action id，是否需要在类型层做 branded type 区分 ActionId vs 其他 Id？当前不做，留给 runtime validation。
